Chapter 11 — Debugging

Failure、Fault 与 Debugging
FailureFaultDebugging
术语含义
Failure运行时可观察到的失败现象 / 症状（symptoms）
Fault代码或设计中的缺陷原因（bug 的根因）
Debugging 的目标是定位并修正 fault；Testing 用于发现 failure。二者目的不同。
考点
“Purpose of debugging” 选 fix faults，不是 fix failures / find faults。
调试过程（Debugging Process）
ReproduceFocusIsolate
常见步骤包括复现、聚焦、隔离等；课程中若出现 “Which is NOT part of debugging process”，需结合选项判断（例如 None of the above 表示所列步骤都属于调试）。
  • 调试与测试目的不同：不是同一件事（False 题常考）。
Bug Report 与 Problem Tracker
BugzillaDuplicateHistory
内容说明
Bugzilla典型的 Bug tracking / 问题跟踪 工具（不是 IDE 或通用 debugger）
Problem history可包含复现步骤等信息；报告可记录 duplicate bug id
Tracking / Reproducing 中的步骤
NeutralizeDisambiguateGeneralizeIsolate
步骤作用（记忆要点）
Neutralize去掉负面、无关或情绪化表述，保留技术事实（“Remove negative information”）
Disambiguate消除描述中的歧义
Generalize在仍能复现的前提下扩大条件（如多种字体仍崩溃）
Isolate缩小范围、明确必要条件（如 “ONLY Windows XP”）
考点
Isolate 与 Generalization 可以协同出现，并非互斥（判断题常为 False）。
Failing-Input 简化：Binary Search & Delta Debugging
Binary SearchDelta Debugging
  • 大输入中找失败行：二分次数约为 log₂(n)（n 为行数）。
  • Binary search 后仍可用 delta debugging 继续缩小。
  • Delta debugging：先尝试少量大块改动，再逐步更多、更小的改动。
Fault Localization 策略
BacktrackingTracingCause Elimination
策略典型场景
Backtracking沿 error propagation chain 回溯；通常不涉及“把指令倒序执行”之类
Tracing断点、断言、print、单步等观察执行
Cause elimination系统排查可能原因；需对程序与常见错误有经验
Summary
Tracing 常用手段包括 print、breakpoint、assertion、single-step 等，题目可考 All of above。
Assertions：调试 vs 测试 & 生产代码
DebugTestProduction
调试中与测试中断言的用法/侧重点可不同（判断题常为 True）。
实践
断言一般不应作为正式生产代码的常规依赖（课程常考 No）。
Design by Contract
PreconditionPostconditionInvariant
  • Postcondition：方法执行后应成立的断言。
  • Invariant：在课程语境下常考“始终为真”的表述（判断题 True）。
  • Loop invariant：在循环前、每次迭代、结束后均需保持（与具体题目选项一致）。
