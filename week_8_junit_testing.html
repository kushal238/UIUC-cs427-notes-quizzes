Week 8 — Software Testing with JUnit

Software Testing 的作用
TestingFunctional RequirementsVerification
Software Testing 用于检验软件功能是否按照需求正确运行，通过执行测试用例并比对预期结果来发现缺陷。
核心定义
Testing = 检查软件功能是否符合 functional requirements 的过程。
小测试优于大测试（Small Tests vs. Big Tests）
Test GranularityAssertionFail Fast
单个大测试中如果前面的 assertion 失败，后续断言通常不会执行，导致定位问题困难；拆成多个小测试更利于排错和维护。
方式行为推荐
Big Test前面失败会阻断后续 assertions不推荐
Small Tests测试独立执行，失败互不影响推荐
考点
“a big test is worse than several small tests” 的关键理由：前一个 assertion 失败会让后续 assertion 不执行。
JUnit 框架核心能力
JUnitTest RunnerAutomationImmediate Feedback
  • Test runners 可以自动执行测试
  • 测试会自检并给出 pass/fail immediate feedback
考点
“Which is correct about JUnit?” 常见正确项是 Both A and B。
assertEquals 方法签名与 delta
assertEqualsdoubledeltaAPI Signature
签名是否有效原因
assertEquals(double expected, double actual, double delta)✓浮点比较需要误差容限
assertEquals(char expected, char actual, double delta)✗char 不需要 delta
assertEquals(long expected, long actual, double delta)✗long 不需要 delta
规则
delta 仅用于 float/double 比较。
Assume 与 AAA 模式
AssumeArrangeActAssert
assumeTrue()/assumeFalse() 用于前置条件判断，不满足时测试会 skip 而不是 fail。
例子
assumeTrue(System.getProperty("os.name").contains("Windows"))：在非 Windows 环境时，后续标记为 line* 的代码不会运行。
JUnit 生命周期注解对比
@Test@BeforeEach@AfterEachLifecycle
注解执行时机用途
@Test标识测试方法执行测试逻辑
@BeforeEach每个测试前准备对象和数据
@AfterEach每个测试后清理状态和资源
@BeforeAll所有测试开始前一次全局初始化
@AfterAll所有测试结束后一次全局释放
易错点
@AfterEach 是“每个测试之后运行”，不是 before each test。
@ParameterizedTest 与参数来源注解
@ParameterizedTest@ValueSource@CsvSource@MethodSource
注解作用
@ParameterizedTest声明该方法为参数化测试
@ValueSource提供简单参数列表
@CsvSource提供多参数 CSV 数据
@MethodSource由工厂方法提供复杂参数
参数化测试参数来源
@ValueSource@CsvSource@MethodSource
来源注解适用场景特点
@ValueSource单参数、简单类型最轻量
@CsvSource多参数组合支持自定义 delimiter
@MethodSource复杂对象参数工厂方法返回 Stream/Iterator/Iterable/array
考点
@MethodSource 工厂方法合法返回类型：Stream、Iterator、Iterable、array。
Implicit Conversion vs Explicit Conversion
Implicit ConversionExplicit ConversionTypeConverter
类型说明
Implicit ConversionJUnit 自动把字符串参数转成目标类型
Explicit Conversion通过自定义 TypeConverter 显式转换
考点
题目问“使用自定义转换器属于哪种转换”时，答案是 Explicit Conversion。
Property-first guideline / Assumption-first guideline
Property-first guidelineAssumption-first guidelineObserver
Guideline（英文）关注点
Property-first guideline列举、制定并测试 observers 的所有可能返回值/属性，确保每种输出都被覆盖。
Assumption-first guideline先明确合法输入与前提假设（哪些输入有效），再设计 Act / Assert。
  • Quiz 里 “listed … all possible returns of observers” → 选 Property-first guideline。
  • Quiz 里涉及 assumeTrue / assumeFalse 的泛化步骤 → Add assumptions（与 Assumption-first 概念相关，但题目问的是 step 名称）。
高频题
两条 guideline 的辨析见上表；non-primitive 对象作参数 → 泛化步骤 Parameterize。
测试泛化（Test Generalization）四步
ParameterizeGeneralize Test OracleAdd AssumptionsCross Test Generalize
步骤核心动作常见 API/技术
Parameterize把固定值替换成参数（含非 primitive 对象）@ParameterizedTest / @MethodSource
Generalize Test Oracle把具体断言推广为性质断言Property-based assertions
Add Assumptions增加输入前提约束assumeTrue / assumeFalse
Cross Test Generalize抽取多个测试的共性逻辑共享 setup/helper
