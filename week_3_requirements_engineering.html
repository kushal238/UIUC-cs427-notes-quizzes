Week 2 — Requirements Engineering

Requirements Engineering 是什么？
Requirements Engineering Real-world Goals Constraints Stakeholders
Requirements Engineering 关注的是软件系统要实现的现实世界目标、软件的功能以及对软件的约束条件，同时负责发掘并协调各方利益相关者的期望。
  • 帮助工程师理解并调整需求 (understand and adjust requirements)
  • 发现 stakeholder 的期望，并管理其预期 (adjust their expectations)
  • 不是负责"设计系统如何实现需求" — 设计属于 Design Phase
考点提示
"Concerned with designing a software system with respect to requirements" 是错误描述。Requirements Engineering 关注发现与定义需求，而非设计实现方案。
为什么要写需求文档？
Manager Developer Tester Support Engineer
需求文档服务于多个角色，各自有不同的使用目的：
角色使用方式合法用途？
Manager根据需求分配资源、管理开发进度✓ 是
Developer按照需求实现系统功能✓ 是
Tester根据需求验证软件是否符合预期✓ 是
Support Engineer决定是否停用系统✗ 否
考点提示
Support Engineer 用需求文档来"决定是否停用系统"不是写需求的理由。停用决策属于业务/运维范畴，与需求规格无直接关系。
Requirement Specification 的常用 Notations
UML User Story Data Flow Diagram Test Plan
需求规格说明书可用多种符号/格式表示，但Test Plan（测试计划）不是需求规格的表示方式。
UML
图形化建模语言，用于结构/行为描述
User Story
敏捷开发中以用户视角描述需求
Data Flow Diagram
描述数据在系统中的流动
Test Plan ✗
测试阶段产物，非需求表示方式
Functional vs. Non-functional Requirements
Functional Requirement Non-functional Requirement Behavior Quality Attribute
需求分为两大类：描述系统做什么（功能性需求）和描述系统做得多好（非功能性需求）。
Functional RequirementNon-functional Requirement
关注点系统应执行的具体行为/功能系统的质量属性、性能约束
特征描述"系统应做什么"描述"系统应做得多好"
典型关键词shall send / shall record / shall display…within N seconds / within N minutes…

典型例题辨析

需求描述类型理由
A text message shall be sent when a customer signs up.Functional描述系统触发的具体行为
The table's full contents shall be displayed within 3 seconds.Non-functional描述性能约束（时间限制）
A new user shall finish ordering within 10 minutes.Non-functional描述可用性约束（时间限制）
Every invoice shall be assigned a unique identifier.Functional描述系统应执行的数据管理行为
The server code shall be written in C.约束性需求技术实现约束，属于广义非功能或约束类
判断技巧
出现 within N seconds/minutes、user-friendly、available 99.9% 等时间/质量限定词 → 通常是 Non-functional Requirement。描述具体的系统动作（send、record、assign）→ 通常是 Functional Requirement。
Mandatory vs. Optional：shall / should / will
shall should will Mandatory Optional
需求文档中，动词的选择决定了需求的强制程度：
词汇强制程度用途
shall强制 (Mandatory)必须实现的需求，不可协商
should建议 (Optional)期望但非强制的需求
will陈述/预期描述预期行为，非强制承诺
例题
"The driver's door ____ be closed once the car is moving." → 答案：shall（安全关键需求，必须强制执行）
Prescriptive vs. Descriptive Statements
Descriptive Statement Prescriptive Statement Domain Fact
需求文档中会出现两种性质截然不同的陈述：
类型含义特征
Descriptive描述现实世界中恒为真的事实（domain fact）无 shall/should，客观陈述
Prescriptive规定系统应做什么（需求/规定）含 shall/should，对系统有约束力
例题辨析
  • "The same person cannot be in two buildings at once." → Descriptive（现实中的物理事实，与软件无关）
  • "A user shall not use the same registration code more than once." → Prescriptive（对系统行为的规定）
Requirements Elicitation（需求启发/收集）
Elicitation Stakeholder Expectations Interview Brainstorming Survey
Elicitation 是发现 stakeholder 期望并管理/调整这些期望的过程，而非告知开发者如何实现，也非验证产品质量。

常见 Elicitation 技术

技术说明属于 Elicitation？
Questionnaires / Surveys / Interviews通过问卷、调查或访谈收集需求✓ 是
Brainstorming & Focus Group集体头脑风暴，聚焦用户群体讨论✓ 是
Early Feedback from Stakeholders原型或早期版本获取反馈✓ 是
User Acceptance Testing (UAT)让用户验收测试软件✗ 否
考点提示
User Acceptance Testing 是 Testing Phase 的活动，用于验证软件是否符合需求，不是 elicitation 技术。Elicitation 发生在需求阶段，目的是发现需求，而非验证实现。
Use Case 的三大组成部分
Use Case Actor Use Case Diagram Test Case
Use Case 需求规格由三个核心组成，Test Case（测试用例）不属于其中。
Use Case
描述系统响应某个动作时发生的事件序列
Use Case Diagram
图形化呈现 Actor 与 Use Case 的关系
Actors
与系统交互的外部角色（人或系统）
Test Cases ✗
属于测试阶段，非 Use Case 组成
Actors：外部角色，不是系统内部
Actor Input Output External
Actor 是与系统交互的外部实体，可以是人、外部系统等。
  • Actor 可以向系统提供 inputs（输入）
  • Actor 可以从系统接收 outputs（输出）
  • Actor 可以同时执行输入和输出操作
  • Actor 在系统外部，不属于系统内部
常见错误
"Actors are inside of the system" → 错误！ Actor 永远处于系统边界之外。
四种 Use Case 形式及用途
Actor-goal List Use Case Brief Casual Use Case Fully Dressed Use Case
形式详细程度主要用途
Actor-goal List最简略帮助决定设计范围 (design scope)
Use Case Brief简短快速描述主要成功场景
Casual Use Case中等，普通段落描述主要成功场景 + 失败处理
Fully Dressed Use Case最详细，有标签分区完整描述所有场景与异常处理
Casual vs. Fully Dressed
两者都强调 "main success scenarios" 并包含失败处理，区别在于格式：Casual 用普通段落，Fully Dressed 有带标签的专属分区。

两者都不是用来告诉开发者"如何实现需求"的——它们描述要做什么，而非怎么做。
考点提示
"Casual and Fully Dressed use cases are used to tell developers how to implement some requirements." → 错误！ Use Case 描述行为需求，不规定实现方式。
Use Case 的写作要求
Event Sequence Complete Functionality Readable Multiple Actors
一个合格的 Use Case 应满足以下条件：
  • 描述系统响应某动作时发生的事件序列 (sequence of events)
  • 描述的功能应从头到尾是一个完整的功能单元（不是片段）
  • 易于阅读，简短但能讲清一个完整故事
  • 一个 Use Case 可以关联多个 Actor（并非限于一个）
常见错误
"A use case cannot be associated with more than one actor." → 错误！ Use Case 可以与多个 Actor 关联。
需求书写中的常见缺陷 (Common Flaws)
Unmeasurability Contradiction Poor Structuring Inadequacy
好的需求应当可度量、无歧义、结构清晰、足够充分。以下是四类常见缺陷：
缺陷类型含义典型表现
Unmeasurability无法量化验证"The system shall be user-friendly."
"The system shall be fast."
Contradiction需求之间相互矛盾两条需求的约束条件互相冲突
Poor Structuring表述混乱，难以理解多个需求混写在一句话里
Inadequacy需求不完整，遗漏重要场景缺少边界条件、异常处理描述
例题
"The book-order process shall be user-friendly." → 典型的 Unmeasurability 缺陷——"user-friendly" 无法被客观测试或度量，应改为具体可测指标，如 "shall be completed within 3 clicks"。
