Week 4 — Software Design and Architecture

Abstraction — Unix File Descriptor 的抽象
Abstraction File Descriptor Unix Interface
Abstraction（抽象）的核心思想：对外暴露接口，隐藏底层实现细节。

在 Unix 中，File Descriptor 是对文件操作的统一抽象——进程通过 File Descriptor 访问文件，无需知道文件的实际名称或存储位置。
  • 进程拿到 File Descriptor（一个整数 handle）后，即可读写文件
  • 底层是本地文件、网络 socket 还是管道，对进程透明
  • 这正是抽象的价值：调用方无需了解实现细节
考点提示
False：在 Unix File Descriptor 抽象下，进程不需要知道文件的实际名称即可访问文件。答案是 False。
Module Cohesion（模块内聚性）
Cohesion Coincidental Logical Temporal Communicational Functional
Cohesion 衡量一个模块内部各元素之间的关联紧密程度。内聚性越高，模块越独立、可维护性越强。

类型定义例子
Coincidental函数被随意打包，无任何逻辑关联把所有杂项函数塞进 utils.c
Logical函数功能类似，但执行不同任务（通过参数选择）一个模块处理所有 I/O 操作
Temporal函数在同一时间段执行（如初始化时）startup() 里初始化所有不相关组件
Communicational函数操作相同的数据对同一记录的读、写、更新
Functional所有函数共同完成单一明确任务calculateTax() 只负责税额计算
考点提示
函数 功能相似（similar functionalities）但执行不同任务 → Logical Cohesion（不是 Coincidental）。
最差的内聚类型是 Coincidental Cohesion。
Pipe-and-Filter Architecture Style
Pipe-and-Filter Filter Pipe Data Flow Unix
数据依次流经多个 Filter（过滤器），每个 Filter 对数据进行变换后通过 Pipe（管道） 传给下一个 Filter，最终输出结果。
  • 每个 Filter 是独立的处理单元，不知道其他 Filter 的存在
  • Filter 之间不共享状态（stateless）
  • 数据单向流动（one direction）
  • Filter 可以并行实现
  • 数据流入即处理（incremental processing）
经典例子
Unix 命令行：cat file.txt | grep "error" | sort | uniq
每个命令是一个 Filter，| 是 Pipe，数据依次流过。
考点提示（Q5）
NOT true： "Filters know each other and share state" — Filter 之间不认识对方、不共享状态，这是 Pipe-and-Filter 的核心约束。
常见 Architecture Styles 比较
风格核心思想典型场景
Pipe-and-Filter数据依次流经独立 Filter 变换编译器、数据处理管道、Unix shell
Layered / Tiered按层次组织，每层只与相邻层交互TCP/IP 协议栈、企业分层架构

Q3 解析
前端接收传感器数据 → 各组件依次变换数据 → 最终渲染可视化 → 这是数据单向流动 + 独立变换的典型模式，答案是 Pipe-and-Filter Style，而非 Layered Style。
Components vs. Connectors
Component Connector Interaction Rules Encapsulation
Component（组件）Connector（连接器）
定义封装计算逻辑与数据的基本单元管理组件之间交互的机制
职责执行具体功能，持有状态定义交互规则，协调通信
例子Service、Module、Filter、LayerPipe、Event bus、RPC call、Protocol
Connector 的三大职能：
  • Interactions among components（组件间交互）
  • Rules that govern those interactions（交互规则）
  • 不包含："Encapsulates computation and/or data" — 这是 Component 的职责
考点提示（Q6）
"Encapsulates computation and/or data" 描述的是 Component，不是 Connector 的功能。Connector 负责交互，不负责封装数据。
MVC Pattern — Model / View / Controller
Model View Controller Separation of Concerns Multiple Views
Model
数据与业务逻辑，与 UI 无关
View
数据的可视化呈现，面向用户
Controller
接收用户输入（gesture events），协调 Model 和 View
  • Controller 接收用户的交互事件（user-gesture events），如点击、键入
  • Model 不依赖 View，修改 Model 不影响整体结构
  • 同一 Model 可以有多个 View（Multiple Views）——同一数据库数据可渲染为折线图、饼图、柱状图
Q8 解析：为什么同一数据可以用多种图表展示？
MVC 的"Ability to provide multiple views"特性：Model（数据库数据）与 View（图表呈现）解耦，同一 Model 可绑定多个不同的 View 实现，互不干扰。
Q7 考点
接收 user-gesture events 的组件是 Controller，不是 View（View 只负责显示）。
Software Modularity（软件模块化）
Modularity Independence Reusability Robustness Compilation
将系统分解为若干独立模块，每个模块封装特定功能，模块之间通过接口通信。
模块化的优势：
  • 模块具有鲁棒性（Robust）——局部错误不会轻易蔓延
  • 模块可以被其他模块复用（A module can use other modules）
  • 模块可以独立编译并存储在库中（separately compiled and stored in a library）
  • 模块之间应相互独立（independent），不是相互依赖
考点提示（Q9）
NOT correct： "Modules are mostly dependent"。
模块化的核心目标是降低耦合、提高独立性。Modules 应该 mostly independent，而非 dependent。
Conway's Law（康威定律）
Conway's Law Team Structure System Architecture Communication
"Organizations which design systems are constrained to produce designs which are copies of the communication structures of those organizations."

即：系统的架构往往镜像了构建它的团队的沟通结构。
  • 编译器前端团队 → 负责 Compiler Front-End 组件
  • 编译器后端团队 → 负责 Compiler Back-End 组件
  • 团队边界 ≈ 系统模块边界
定律内容领域
Conway's Law系统架构 ≈ 团队沟通结构软件工程 / 组织管理
Brooks' Law向延期项目增人反而更慢项目管理
Moore's Law芯片晶体管数量每两年翻倍硬件 / 半导体
Bell's Law每十年出现新的计算机类别计算机历史
Q4 解析
公司内多个团队各自负责编译器的一个组件（前端团队、后端团队），团队组织结构映射到了系统组件结构——这正是 Conway's Law 的体现。
