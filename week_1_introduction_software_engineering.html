Week 1 - Introduction to Software Engineering

Software Development Life Cycle (SDLC)
SDLC Requirements Design Implementation Testing Maintenance
SDLC 是软件从无到有、从发布到维护的完整生命周期框架，定义了开发过程中每个阶段的任务与目标。
阶段核心任务产出物
Requirements Phase收集、分析用户需求需求规格说明书 (SRS)
Design Phase规划架构、模块、接口设计文档、UML 图
Implementation Phase编码开发源代码
Testing Phase验证功能与质量测试报告
Maintenance Phase修复 bug、迭代更新补丁、新版本
考点提示
Evaluation Phase 不属于 SDLC 标准阶段。 常见干扰项，注意区分。
The Trinity of Software Project（软件项目三元组）
Software System Software Users Software Development Process Trinity
软件项目由三个核心要素构成，缺少任一要素都不完整。三者相互依存、共同决定项目成败。
Software System
最终交付的软件产品本身
Software Users
使用该系统的人群
Software Dev. Process
构建软件所用的方法与流程
考点提示
Software Design 不在三元组内。 Design 是开发过程中的一个阶段，而非独立的三元组成员。
Information Hiding（信息隐藏）
Information Hiding Encapsulation Interface Modularity
将组件的内部实现细节隐藏起来，只对外暴露接口 (Interface)。当内部设计变更时，外部调用方代码无需修改。
  • 修改内部逻辑 → 外部代码不受影响
  • 提升模块独立性，降低耦合度 (coupling)
  • 是 OOP 中 Encapsulation（封装）的核心思想
概念重点
Information Hiding隐藏内部决策，对外只暴露接口
Divide and Conquer把大问题拆分成小问题分别解决
Inheritance子类复用父类属性与方法（OOP 机制）
Project Management计划、组织、控制项目执行
Software 的核心特征：Malleable（可塑性）
Malleable Complex Evolving Team-based
软件最显著的特征是高度可塑性 (Malleable)——可以被轻易修改、扩展、重构，这是与硬件最大的不同。
特征说明正确？
Malleable易于修改与调整✓ 正确
Simple现实中软件高度复杂✗ 错误
Constant软件持续演进，非恒定不变✗ 错误
Single-person现实软件通常需要团队协作✗ 错误
Brooks' Law — 增人不增速
Brooks' Law Software Crisis Communication Overhead Onboarding
"Adding manpower to a late software project makes it later." — Fred Brooks，《The Mythical Man-Month》
  • 新成员需要上手时间 (onboarding)，短期内不产出
  • 人员增加 → 沟通路径指数级增长 → communication overhead 急剧上升
  • 老成员被迫分心辅导新人 → 整体效率下降
一句话总结
增加团队人数不能解决 Software Crisis，反而可能使情况更糟。
Project Stakeholders（项目利益相关方）
Stakeholder Customer User Developer Tester Designer
所有对软件项目有利益关联或影响的人，统称为 Stakeholders。范围远比"开发者"更广。
角色描述
Customer付费购买/委托开发软件的人或组织
User实际使用软件的人（可与 Customer 不同）
Designer负责系统架构与界面设计
Tester验证软件质量与功能正确性
Support Engineer负责部署、运维与用户支持
Project Manager协调资源、进度与风险
Users ≠ Customers
  • Customer：出钱的人，如企业采购方
  • User：用软件的人，如企业员工
  • 两者可以是同一人，也可以完全不同（常见于 B2B 场景）
例子
公司 IT 部门购买了一款 CRM 软件（Customer = IT 部门），实际使用的是销售团队（Users = 销售人员）。
Software Engineering vs. Programming
Software Engineering Programming Multi-person Multi-version Long-lived Systems
Programming（编程）Software Engineering（软件工程）
规模个人或小脚本Multi-person 团队协作
版本通常单一版本Multi-version，持续演进
寿命短期、临时长期维护（Long-lived）
关注点让代码跑起来流程、设计、质量、协作
产出可运行程序可维护的软件系统
  • SE 包含编程，但远不止编程——还涉及需求、设计、测试、维护等全流程
  • SE 的核心挑战是在多人协作、长期演进的背景下保持质量与一致性
  • SE 关注 "how to build software well at scale"，而不仅仅是 "how to write code"
软件工程师的关键属性
除了技术能力，还需具备 ability to efficiently/effectively interact and work in a team——团队协作能力是软件工程师区别于单纯程序员的核心素质。
Competing Goals of Software Project（竞争性目标）
Time Cost Quality Iron Triangle Trade-off
软件项目的三大竞争性目标相互制约，通常无法同时最优——即项目管理中的 "Iron Triangle"（铁三角）。
Time（时间）
开发者用于开发的时间与项目周期
Cost（成本）
客户支付给团队的开发费用
Quality（质量）
软件的功能完整性与可靠性
  • 想要更快交付 → 需要增加成本 或 降低质量
  • 想要更便宜 → 需要延长时间 或 降低质量
  • 想要更高质量 → 需要更多时间 或 更高成本
不属于竞争性目标的选项：
考点提示
Market share（市场占有率）是业务/商业指标，不属于软件项目的 competing goals。Time、Cost、Quality 才是三大核心竞争目标。
