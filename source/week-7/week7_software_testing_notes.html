核心概念
Testing 的目的与局限
Testing Code Quality Fault Detection Limitation
Testing 的作用是发现问题、提升质量、作为文档，但它有一个根本局限：无法保证代码完全正确。
Testing 的目的说明是否正确
Improve code quality通过测试发现并修复缺陷，提升整体质量✓
Spot code faults识别代码中存在的 bug 和错误逻辑✓
Document codeTest cases 本身可作为系统行为的活文档✓
Guarantee fully correct输入空间无限大，无法穷举所有情况✗ 不可能
考点提示
Testing 不能保证代码"完全正确"——这是最常考的干扰项。即使通过所有测试，也可能存在未被覆盖的边界情况。
实践原则
测试的时机与规模
TDD Test-first Test Code Volume UI Testing
测试可以在开发的任意阶段编写，并且测试代码量可以超过生产代码量。
  • Before — 写代码前先写测试（TDD / BDD），驱动设计
  • During — 开发中同步补充测试，保证新功能有覆盖
  • After — 代码完成后补测试，常见于回归测试和 bug 修复
  • 成熟项目中，test code 量可以超过 source code，这是正常现象
  • User Interface 也可以被测试，UI 自动化测试在设计稳定后构建
关于谁来写测试
测试不只是 QA 的职责。开发者、领域专家甚至用户都可以参与编写 test cases，多方参与能减少盲区偏差。
认知偏差
Developer Bias in Testing
Developer Bias QA Independent Testing
开发者测试自己的代码时，往往存在认知偏差（bias）——倾向于测试"认为正确"的路径，而忽略边界情况。
  • 开发者即使完全理解实现，也不能保证发现所有 faults
  • 对自己写的代码有先入为主的假设，难以从外部视角测试
  • 鼓励更多人参与写测试——独立视角能发现开发者看不到的问题
  • QA 工程师是测试专家，但并非唯一应该写 test cases 的人
测试流程
四种 Test Activity 类型
Test Design Test Automation Test Execution Test Evaluation Test Implementation ✗
Activity核心任务所需能力
Test Design设计测试用例，确定测试值与预期结果深入的领域与代码知识
Test Automation将测试值嵌入可执行脚本，实现自动化运行编程能力，理解测试框架
Test Execution实际运行测试，收集结果执行环境配置能力
Test Evaluation分析测试结果，判断通过/失败，评估覆盖率结果分析与报告能力
考点提示
"Test Implementation" 不是标准 activity 类型，是最常考的干扰项。记住四类：Design → Automation → Execution → Evaluation。
自动化 vs 手工
Manual Testing vs. Automated Testing
Automated Testing Manual Testing Regression Testing
Manual TestingAutomated Testing
适用场景探索性测试、一次性测试、UI 原型期回归测试、高频重复执行的测试
Bug 发现擅长创意性发现，探索边界不一定比手工发现更多 bug
维护成本低（无需维护脚本）高（脚本需随代码更新）
UI 测试时机设计阶段可手动验证界面设计稳定后再自动化
常见错误认知
自动化测试并不总是比手工测试发现更多 bug。对于不需要重复执行的测试，手工完成即可，自动化反而浪费资源。
回归测试原则
当 regression tests 需要频繁执行时，优先选择自动化——这是自动化的核心价值所在。
文档与理解
Test Cases 作为活文档
Test Cases Living Documentation Requirements Understanding
编写 test cases 的过程本身，能帮助开发者深化对需求的理解。Test cases 同时也是系统行为的活文档。
  • Test cases 不只是代码的附属品，它们描述系统预期行为
  • 写测试时需要明确输入、输出和边界——这本身就是对需求的梳理
  • Test cases 可以作为文档被团队成员阅读和参考
  • 测试不一定以代码形式存在（也可以是手工测试计划文档）
测试策略对比
Black-box vs. White-box Testing
Black-box White-box Structural Testing Functional Testing
Black-box TestingWhite-box Testing
视角从外部，不知道内部实现从内部，完整了解代码结构
别名Functional / Specification-basedStructural / Code-based

考点提示
White-box testing 并不简单，需要深入了解编程和实现细节。Black-box testing 在发现 extra logic faults 方面较弱（因为看不到内部多余代码）。
Black-box 核心局限
无法穷举所有测试输入
Exhaustive Testing Input Space Infeasible
Black-box testing 面临的根本挑战：输入空间几乎无限大，无法对所有可能输入进行穷举测试，因此必须选取代表性样本。
  • 即使是简单函数，合法输入组合也可能达到数十亿种
  • 解决方案：使用 Equivalence Class Partitioning 和 Boundary Value Analysis 等策略缩减测试集
  • 这也是为什么 testing 无法"保证代码完全正确"的底层原因
White-box 核心局限
维护成本与测试方法的迁移
Maintenance Cost Equivalence Class Boundary Value
White-box 测试的脚本与内部实现深度绑定，一旦实现频繁变更，维护脚本的成本极高。
重要补充
Equivalence Class Partitioning 和 Boundary Value Analysis 不只是 Black-box 技法——它们同样可以应用于 White-box testing，属于通用测试设计方法。
缺陷分析模型
PIE Model — 故障传播链
Execution Infection Propagation Failure Observation
PIE 模型描述了一个 failure 被观察到所需的三个必要条件，缺少任何一个都不会产生可见的 failure。
P — Execution
包含 fault 的代码位置必须被执行到
I — Infection
执行后程序状态必须变为不正确（infected）
E — Propagation
被污染的状态必须传播到可观察的输出
考点提示
三个条件都是 必要条件。"infected state must NOT propagate" 是错误表述——Propagation 必须发生，failure 才能被观察到。
概念辨析
Error vs. Fault vs. Failure
Error Fault Failure
概念定义对应层面
Error人（开发者）犯的错误，导致代码写错人的行为层面
Fault代码中实际存在的缺陷（bug）代码/静态层面
Failure程序运行时表现出的错误行为运行时/可观察层面
例子解析
开发者本想调用 Math.log(5) 求自然对数，却写成了求 5³ 返回 125——这个人为的书写错误是 Error；代码中错误的逻辑是 Fault；程序运行时返回 125 而非 ln(5) 是 Failure。
测试度量
Code Coverage Metrics
Statement Coverage Branch Coverage Method Coverage Parameter Coverage ✗
Coverage metrics 衡量测试对代码结构的覆盖程度，是评估测试充分性的核心指标。
Coverage 类型度量对象是否标准
Statement Coverage每条可执行语句是否被执行✓ 标准
Branch Coverage每个条件分支（true/false）是否都被走到✓ 标准
Method Coverage每个方法/函数是否被调用过✓ 标准
Condition Coverage复合条件中每个子条件的真/假情况✓ 标准
Parameter Coverage——✗ 不是标准指标
考点提示
"Parameter coverage" 不是标准覆盖率指标。标准指标围绕代码执行结构（语句、分支、方法、条件）定义，参数不构成独立的覆盖维度。
Black-box 技法
Equivalence Class Partitioning vs. Boundary Value Analysis
Equivalence Class Partitioning Boundary Value Analysis Decision Table Testing Use Case Testing
技法核心思想记忆关键词
Equivalence Class Partitioning将输入条件分组，同组内的输入行为相同，每组只需测一个代表值分组、代表值
Boundary Value Analysis聚焦各分组边界值（最小值、最大值、边界±1）进行测试边界、±1
Decision Table Testing列举条件组合与对应动作，覆盖多条件逻辑条件矩阵
Use Case Testing基于用户使用场景设计测试，从用户角度验证功能用户场景
核心区别
ECP 关注"把输入分成几组"；BVA 关注"在分组的边界处测试"。两者常配合使用——先用 ECP 划定等价类，再用 BVA 在边界处加密测试。
