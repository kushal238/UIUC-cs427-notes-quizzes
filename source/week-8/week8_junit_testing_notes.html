Software Engineering — Week 8 笔记

Module 8 Software Testing with JUnit Coursera
概念
Software Testing 的作用
Testing Functional Requirements Verification
Software Testing 用于检验软件的各项功能是否按照需求规格正确运行。通过设计测试用例执行程序，观察输出是否符合预期，从而发现缺陷。
核心定义
Testing = 检查软件功能是否符合 functional requirements 的过程。
设计原则
小测试优于大测试（Small Tests vs. Big Tests）
Test Granularity Assertion Fail Fast
将测试拆分为多个小测试，而不是写一个包含大量 assertions 的大测试。原因在于：如果前一个 assertion 失败，后续的 assertions 不会被执行，导致问题定位困难。
方式行为推荐？
Big Test（单一大测试）前面 assertion 失败 → 后续全部跳过不推荐
Small Tests（多个小测试）每个测试独立运行，失败互不影响推荐
考点
大测试的缺点：preceding assertion 失败会导致 later assertions 不被执行。
概念
JUnit 框架核心能力
JUnit Test Runner Automation Immediate Feedback
JUnit 是 Java 的主流单元测试框架，提供两大核心能力：
  • Test Runners：自动运行测试用例，无需手动触发
  • Self-checking：测试自动判断 pass/fail，并提供即时反馈 (immediate feedback)
考点
JUnit 同时具备 test runner 和 自动检查结果两项能力，答案选 "Both A and B"。
API
assertEquals 方法签名
assertEquals delta double API Signature
比较浮点数时需要提供误差容限参数 delta，因为浮点数存在精度问题。有效签名：
签名是否有效原因
assertEquals(double, double, double delta)✓ 有效double 需要 delta 参数
assertEquals(char, char, double delta)✗ 无效char 是整型，不需要 delta
assertEquals(long, long, double delta)✗ 无效long 是整型，不需要 delta
规则
delta 参数只用于浮点类型（float / double），整型比较不需要 delta。
Assume vs Assert
Assume — 测试前提条件
Assume Arrange Act Assert AAA Pattern
JUnit 测试通常遵循 AAA 模式（Arrange → Act → Assert）。Assume 用于在测试执行前检查前提条件——如果条件不满足，测试被跳过而非失败。
步骤作用API 示例
Assume检查测试前提（条件不满足则 skip）assumeTrue(condition)
Arrange准备测试数据与对象变量赋值、对象创建
Act调用被测方法result = obj.method()
Assert验证结果是否符合预期assertEquals(expected, actual)
例子
assumeTrue(System.getProperty("os.name").contains("Windows")) — 仅在 Windows 系统上运行该测试，其他系统直接 skip。
考点：如果 OS 不是 Windows，该行后的代码（line*）不会被执行。
核心 Annotations
JUnit 生命周期注解对比
@Test @BeforeEach @AfterEach Lifecycle
注解执行时机典型用途
@Test标识测试方法，由 runner 调用每个测试用例
@BeforeEach每个 @Test 方法执行前运行初始化数据、创建对象
@AfterEach每个 @Test 方法执行后运行清理资源、重置状态
@BeforeAll所有测试方法开始前运行一次耗时的全局初始化
@AfterAll所有测试方法结束后运行一次全局资源释放
易错点
@AfterEach 在每个测试之后运行，不是之前。
"Annotating with @AfterEach causes that method to run before each test" → False。
参数化注解
@ParameterizedTest 与 @ValueSource
@ParameterizedTest @ValueSource Arguments Source
参数化测试需要两个注解配合使用：@ParameterizedTest 标识该方法为参数化测试，@ValueSource（或其他 source 注解）提供参数来源。
注解职责
@ParameterizedTest声明方法为参数化测试（替代 @Test）
@ValueSource提供简单类型的参数列表
@CsvSource提供 CSV 格式的多参数组合
@MethodSource指定工厂方法提供复杂参数
考点
为参数化测试的每次调用提供参数来源，使用 @ValueSource（或其他 source 注解）。
参数来源
三种参数来源方式
@CsvSource @MethodSource CSV Delimiter Stream / Iterator
来源注解适用场景返回类型
@ValueSource单个简单参数（int、String 等）字面量数组
@CsvSource多参数组合，CSV 格式；可指定 delimiter字符串（自动解析）
@MethodSource复杂对象参数Stream / Iterator / Iterable / Array
@CsvSource 支持通过 delimiter 属性指定分隔符，可以传递多个参数。
@MethodSource 指向一个工厂方法，该方法返回类型可以是 Stream、Iterator、Iterable 或 array of element sets。
考点
@MethodSource 工厂方法可接受的返回类型：Stream / Iterator / Iterable / 数组，四者均合法。
类型转换
Implicit vs. Explicit Conversion
Implicit Conversion Explicit Conversion TypeConverter
参数化测试中，JUnit 支持两种参数类型转换方式：
类型说明示例
Implicit ConversionJUnit 自动将 String 转换为目标类型String → int / enum 等
Explicit Conversion使用自定义 TypeConverter 显式转换自定义对象类型
考点
当测试方法参数类型需要使用自定义 TypeConverter 转换时，属于 Explicit Conversion。
测试策略
测试泛化的四个步骤
Parameterize Generalize Test Oracle Add Assumptions Cross Test Generalize
将一个具体测试用例泛化为更通用的测试，分四个步骤：
步骤说明关键 API / 技术
Parameterize将字面量参数替换为可变参数；非基本类型对象作为参数时使用此步@ParameterizedTest、@MethodSource（非 primitive 对象）
Generalize Test Oracle将具体期望值替换为通用属性或表达式Property-based assertion
Add Assumptions添加前提条件，排除无效输入assumeTrue() / assumeFalse()
Cross Test Generalize跨测试方法提取共同逻辑，减少重复@BeforeEach、工具方法
测试指导原则
Property-first Guideline
Property-first Assumption-first Observer Return Value
Property-first Guideline：列举、制定并测试被测对象（observer）所有可能的返回值，确保每种输出属性都被覆盖到。
Assumption-first Guideline：先明确测试的前提假设（assumptions），再设计测试用例——先定义哪些输入是合法的，再基于合法输入编写断言。
指导原则关注点
Property-first枚举所有 observers 的可能返回值并测试
Assumption-first先确定有效输入范围（assumptions），再编写断言
考点
"列举并测试所有 observer 可能的返回值" → Property-first guideline。
使用 assumeTrue / assumeFalse → Add Assumptions 步骤。
步骤辨析
Parameterize 步骤 — 非基本类型参数
Non-primitive Object @MethodSource Parameterize
当测试参数是非基本类型对象（non-primitive objects）时，对应的泛化步骤是 Parameterize。此时通常使用 @MethodSource 来提供对象参数，因为 @ValueSource 仅支持简单类型。
  • 基本类型（int / String / double）→ @ValueSource 或 @CsvSource
  • 自定义对象（非 primitive）→ @MethodSource 返回 Stream<Arguments>
