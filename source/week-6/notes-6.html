核心概念
Design Patterns — 什么是设计模式？
Design Pattern Reusable Template Recurring Problem GoF
Design Patterns 是针对软件开发中反复出现的问题所提炼出的可复用解决方案模板。由"Gang of Four"（GoF）在《Design Patterns》一书中系统化整理，共 23 种经典模式。
  • 不是具体代码，而是解决问题的思路与结构
  • 提升代码的可读性、灵活性与可维护性
  • 一个 class 可以同时参与多个 Design Pattern
  • 使用统一词汇，方便团队沟通
考点提示
"A class cannot participate in more than one Design pattern" — 错误！一个类完全可以同时实现多个模式的角色。

Creational Patterns
创建型模式 — 关注对象如何被创建
Creational Object Creation Instantiation Flexibility
创建型模式将对象的创建逻辑与使用逻辑解耦，使系统不依赖于具体类的实例化方式，从而提升灵活性与可扩展性。
模式核心思想
Singleton确保类只有一个实例，并提供全局访问点
Factory Method定义创建对象的接口，由子类决定实例化哪个类
Abstract Factory创建一组相关对象的工厂接口，不指定具体类
Builder分步构建复杂对象，同样的构建过程可生成不同表示
Prototype通过克隆（clone）现有对象来创建新对象
Structural Patterns
结构型模式 — 关注类与对象如何组合
Structural Composition Interface Composite
结构型模式描述如何将类或对象组合成更大的结构，同时保持结构的灵活与高效。其中 Composite Pattern 是本模块重点。
Composite Pattern 深入
Composite 模式让单个对象（Leaf）与对象组合（Composite）能被客户端以统一方式处理，形成树形结构。
Component
所有组件的抽象接口，定义公共操作
Leaf
叶节点，无子节点，实现 Component 所有方法
Composite
容器节点，持有子组件，实现子节点管理方法
关键区分
Component 接口只声明所有组件共有的操作（如 render、resize）；
子节点管理方法（add / remove / getChild）属于 Composite 类，而非 Component 接口。
典型场景
3D 对象库：单个 3D 物体（Leaf）和物体组合（Composite）都需要支持 resize、rotation 等相同变换 → 使用 Composite Pattern，统一对待个体与群体。
Behavioral Patterns
行为型模式 — 关注对象间的交互与职责分配
Behavioral Interaction Responsibility Algorithm
行为型模式描述类或对象之间如何交互与分配职责，重点在于算法的封装、对象间的通信以及运行时行为的灵活切换。
模式核心思想关键词
Observer主题状态变化时，自动通知所有订阅者publish-subscribe, 1对多通知
Strategy封装一族算法，使其可在运行时互换runtime 切换, 算法族
Iterator提供顺序访问集合元素的方式，不暴露内部结构遍历, 集合访问
Interpreter定义语言的文法，用类表示每条规则AST, Context, 语法树
Visitor在不修改类的前提下，为其添加新操作操作扩展, 双重分派
State对象行为随内部状态变化而变化状态机, 状态驱动行为
Behavioral — Observer
Observer Pattern（观察者模式）
Observer Subject Notify Decouple 1对多
Observer 模式将对象状态的持有（Subject）与状态的展示/响应（Observer）解耦。Subject 状态改变时，自动通知所有注册的 Observer。
  • 支持多个 Observer 同时监听同一 Subject（1对多）
  • Subject 的状态可以随时改变，改变后推送通知
  • 核心是将状态显示与状态本身分离（separates the display of object state from the object itself）
辨析
Observer 是 "display of state vs. state object" 的分离，不是 "display vs. display implementation" 的分离（后者更接近 MVC/Strategy 的描述）。
Behavioral — Strategy
Strategy Pattern（策略模式）
Strategy Algorithm Family Runtime Switch Encapsulation Interchangeable
Strategy 模式定义一族算法，将每个算法封装成独立的类，使它们可以在运行时相互替换，而不影响使用算法的客户端代码。
  • 将算法的使用与算法的实现完全解耦
  • Context 类持有 Strategy 引用，可随时切换
  • 避免大量 if-else / switch 判断算法分支
典型场景
对冲基金需要在交易过程中实时切换不同的交易策略 → Strategy Pattern，将每种策略封装为独立对象，运行时按需替换。
Behavioral — Interpreter
Interpreter Pattern（解释器模式）
Interpreter Context AbstractExpression TerminalExpression NonTerminalExpression AST
Interpreter 模式为语言定义文法，并用类层次结构表示该文法。通过构建抽象语法树（AST）来解释（执行）语言中的句子。
角色职责
AbstractExpression声明 interpret(Context) 接口
TerminalExpression叶节点表达式，无子节点，可直接解释
NonTerminalExpression内部节点，维护子 Expression 列表，递归解释
Context携带解释过程中的全局信息（非树节点）
高频考点
AST 由 Expression 实例构成，不是由 Context 实例构成。
Context 是解释时传递的共享状态参数，不参与树的构建。

模式核心目的触发场景关键词
Observer状态变化 → 自动通知多个订阅者事件广播、publish-subscribe、通知

Iterator顺序遍历集合，不暴露内部结构遍历集合元素
Visitor不修改类，为其新增操作对现有类添加行为、双重分派
Interpreter解释语言/表达式文法、AST、语言解析
Composite Pattern 细节
Composite 三角色职责边界
Component Leaf Composite
角色实现公共操作？管理子节点？有子节点？
Component（接口）声明否否
Leaf实现否否
Composite实现是（add/remove）是
高频错误
"Component interface includes methods to manipulate children" — 错！子节点管理方法只在 Composite 类中，Component 接口不包含。

关于 Strategy vs. Visitor
两者都可以"外部化"行为，但：Strategy 替换算法本身（整体行为切换）；Visitor 在不改变对象结构的前提下新增操作（操作扩展）。
