1. Testing 的目的与局限
▾
4 个核心目的
  • 提升代码质量（Improve code quality）
  • 发现代码缺陷（Spot code faults）
  • 充当活文档（Document code）
  • 辅助理解需求
不能保证完全正确 ✗
  • 输入空间几乎无限大
  • 任何有限测试集都无法穷举
  • "guarantee fully correct" 是错误目的
任意阶段均可写测试
  • Before — TDD/BDD，驱动设计
  • During — 同步补充，保证覆盖
  • After — 回归测试、修复 bug
  • All of above ✓ 是正确答案
测试代码规模
  • 测试代码量可以超过源码量
  • UI 界面同样可以被测试
  • 测试不一定以代码形式存在
  • 测试无法"总是发现所有 bug"
2. Test Activities 四类
▾
Test Design
设计测试用例，确定输入值与预期输出。
  • 需要深入的领域与代码知识
  • 并非"只需少量知识"
Test Automation
将测试值嵌入可执行脚本，实现自动运行。
  • 有用但编写过程可能单调重复
  • 这一点陈述是正确的
Test Execution
实际运行测试脚本，收集结果。
Test Evaluation
分析测试结果，判断通过/失败，评估覆盖率。
Test Implementation ✗ 非标准类型
不属于标准 test activity 类型，是高频考点干扰项。
Manual vs. Automated
  • Regression 频繁 → 优先自动化
  • UI 测试 → 界面稳定后再自动化
  • 自动化 ≠ 总是发现更多 bug ✗
  • 不需重复执行 → 手工完成即可
谁来写测试
  • 开发者对自己代码有认知偏差
  • 鼓励多方参与：开发者、QA、领域专家
  • "只有 QA 才能写测试" ✗ 错误
  • Test cases 编写过程帮助理解需求
3. Black-box vs. White-box Testing
▾
Black-box Testing
  • 从外部视角，不了解内部实现
  • 别名：Functional / Specification-based
  • 无法穷举所有输入 ← 根本局限
  • 弱于发现 extra logic faults
  • 不适合发现 out-of-bounds 等底层错误
White-box Testing
  • 从内部视角，完整了解代码结构
  • 别名：Structural / Code-based
  • 需要深入编程与实现知识（并不简单 ✗）
  • 实现频繁变动时维护成本极高
  • 适合发现 extra logic、dead code
通用原则
  • ECP / BVA 两种方法均可用于 Black 和 White-box
  • 不是 Black-box 专属技法
4. PIE 模型 — 故障传播链
▾
P — Execution（执行）
包含 fault 的代码位置必须被执行到。未执行 → 不可能触发 failure。
I — Infection（感染）
执行后程序状态必须变为不正确（infected state）。状态正确 → failure 不可见。
E — Propagation（传播）
被感染的状态必须传播到可观察的输出。
考点："infected state must NOT propagate" ✗ 错误表述，propagation 是必要条件。
Error（人为失误）
开发者做出的错误决策，导致代码写错。
例：本应写 Math.log(5)，写成了求 5³。
Fault（代码缺陷）
代码中实际存在的 bug，静态存在于源码。
Failure（运行失败）
程序运行时可观察到的错误行为，如返回 125 而非 ln(5)。
5. Coverage Metrics & 测试技法
▾
标准 Coverage 指标
  • Statement Coverage — 语句覆盖
  • Branch Coverage — 分支覆盖
  • Method / Function Coverage
  • Condition Coverage — 条件覆盖
Parameter Coverage ✗ 非标准
不是标准覆盖率指标。参数不构成独立的执行结构，这是高频干扰项。
Equivalence Class Partitioning (ECP)
  • 将输入条件分组（等价类）
  • 同一组内所有输入行为相同
  • 每组取一个代表值测试即可
  • 关键词：分组 · 代表值
Boundary Value Analysis (BVA)
  • 聚焦等价类的边界值
  • 测试最小值、最大值、边界 ±1
  • 关键词：边界 · ±1
  • 常与 ECP 配合使用
Decision Table Testing
列举多个条件的组合与对应动作，适用于复杂条件逻辑。
Use Case Testing
基于用户使用场景设计测试，从用户视角验证功能正确性。
考点速记
  • Testing 能保证代码完全正确 ✗
  • Test Implementation 是标准类型 ✗
  • Parameter Coverage 是标准指标 ✗
  • PIE 中 Propagation 不必要 ✗
  • 自动化总比手工发现更多 bug ✗
  • 只有 QA 才能写 test cases ✗
